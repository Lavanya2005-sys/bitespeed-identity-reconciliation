# 🔗 Bitespeed Backend Task: Identity Reconciliation

> **A web service that identifies and keeps track of a customer's identity across multiple purchases, even when different contact information (email/phone) is used.**

## 📖 Problem Statement

FluxKart.com uses **Bitespeed** to collect contact details from shoppers for a personalised customer experience. However, customers like the eccentric Dr. Emmett Brown use different email addresses and phone numbers for each purchase to avoid drawing attention.

**Challenge:** Bitespeed needs a way to link different orders made with different contact information to the **same person**.

### How Identity Linking Works

- Contact rows are **linked** if they share either an `email` or `phoneNumber`
- The **oldest** contact in a linked group is treated as `"primary"`
- All other contacts in the group are `"secondary"` and point to the primary via `linkedId`
- When a new request reveals that two previously separate primary contacts belong to the same person, the **older** primary stays primary and the **newer** one is converted to secondary

## 🗄️ Database Schema

The service uses a single table named `Contact`:

```sql
CREATE TABLE Contact (
    id              INTEGER PRIMARY KEY AUTOINCREMENT,
    phoneNumber     TEXT,
    email           TEXT,
    linkedId        INTEGER,          -- ID of the primary Contact this is linked to
    linkPrecedence  TEXT NOT NULL,     -- "primary" or "secondary"
    createdAt       DATETIME NOT NULL,
    updatedAt       DATETIME NOT NULL,
    deletedAt       DATETIME,         -- Soft delete support
    FOREIGN KEY (linkedId) REFERENCES Contact(id)
);
```

**Indexes** are created on `email`, `phoneNumber`, and `linkedId` for fast lookups.

> 💡 The SQLite database file (`contacts.db`) is auto-created in the project root when the server starts for the first time. It is excluded from version control via `.gitignore`.

## 🔌 API Reference

### `POST /identify`

The core endpoint for identity reconciliation.

**URL:** `/identify`
**Method:** `POST`
**Content-Type:** `application/json`

#### Request Body

```json
{
  "email": "string (optional)",
  "phoneNumber": "string or number (optional)"
}
```

> ⚠️ At least one of `email` or `phoneNumber` **must** be provided.

#### Response Format (200 OK)

```json
{
  "contact": {
    "primaryContatctId": number,
    "emails": ["string"],
    "phoneNumbers": ["string"],
    "secondaryContactIds": [number]
  }
}
```

| Field | Description |
|-------|-------------|
| `primaryContatctId` | ID of the primary contact in the linked group |
| `emails` | All unique emails in the group (primary's email comes first) |
| `phoneNumbers` | All unique phone numbers in the group (primary's phone comes first) |
| `secondaryContactIds` | IDs of all secondary contacts linked to the primary |

#### Error Response (400)

```json
{
  "error": "At least one of email or phoneNumber must be provided"
}
```

### `GET /`

Health check endpoint.

**Response:**
```json
{
  "message": "Bitespeed Identity Reconciliation Service",
  "status": "running",
  "endpoint": "POST /identify"
}
```

## 🧪 Example Scenarios (Step by Step)

### Scenario 1: Brand New Customer

**Database:** Empty

**Request:**
```json
{ "email": "lorraine@hillvalley.edu", "phoneNumber": "123456" }
```

**What happens:** No matching contacts found → creates a new **primary** contact.

**Database After:**
| id | phoneNumber | email | linkedId | linkPrecedence |
|----|-------------|-------|----------|----------------|
| 1 | 123456 | lorraine@hillvalley.edu | null | primary |

**Response:**
```json
{
  "contact": {
    "primaryContatctId": 1,
    "emails": ["lorraine@hillvalley.edu"],
    "phoneNumbers": ["123456"],
    "secondaryContactIds": []
  }
}
```

---

### Scenario 2: Returning Customer with New Info

**Database:** Contains contact id=1 from Scenario 1.

**Request:**
```json
{ "email": "mcfly@hillvalley.edu", "phoneNumber": "123456" }
```

**What happens:** Phone `123456` matches existing contact id=1, but email `mcfly@hillvalley.edu` is **new** → creates a **secondary** contact linked to the primary.

**Database After:**
| id | phoneNumber | email | linkedId | linkPrecedence |
|----|-------------|-------|----------|----------------|
| 1 | 123456 | lorraine@hillvalley.edu | null | primary |
| 2 | 123456 | mcfly@hillvalley.edu | 1 | secondary |

**Response:**
```json
{
  "contact": {
    "primaryContatctId": 1,
    "emails": ["lorraine@hillvalley.edu", "mcfly@hillvalley.edu"],
    "phoneNumbers": ["123456"],
    "secondaryContactIds": [2]
  }
}
```

> 💡 **All of these requests will also return the same response above:**
> - `{ "phoneNumber": "123456" }` — matches by phone
> - `{ "email": "lorraine@hillvalley.edu" }` — matches by primary email
> - `{ "email": "mcfly@hillvalley.edu" }` — matches by secondary email

---

### Scenario 3: Merging Two Primary Contacts

**Database:**
| id | phoneNumber | email | linkedId | linkPrecedence | createdAt |
|----|-------------|-------|----------|----------------|-----------|
| 11 | 919191 | george@hillvalley.edu | null | primary | 2023-04-11 |
| 27 | 717171 | biffsucks@hillvalley.edu | null | primary | 2023-04-21 |

These are two **separate** primary contacts with no shared info.

**Request:**
```json
{ "email": "george@hillvalley.edu", "phoneNumber": "717171" }
```

**What happens:**
1. Email `george@hillvalley.edu` matches contact **id=11**
2. Phone `717171` matches contact **id=27**
3. These belong to **two different primary groups** → they must be **merged**
4. Contact id=11 is **older** → stays as **primary**
5. Contact id=27 is **newer** → converted to **secondary** (linked to id=11)

**Database After:**
| id | phoneNumber | email | linkedId | linkPrecedence |
|----|-------------|-------|----------|----------------|
| 11 | 919191 | george@hillvalley.edu | null | primary |
| 27 | 717171 | biffsucks@hillvalley.edu | 11 | **secondary** |

**Response:**
```json
{
  "contact": {
    "primaryContatctId": 11,
    "emails": ["george@hillvalley.edu", "biffsucks@hillvalley.edu"],
    "phoneNumbers": ["919191", "717171"],
    "secondaryContactIds": [27]
  }
}
```

## ⚙️ Core Logic Explained

The `identify()` function in `src/service.ts` handles three main cases:

```
┌─────────────────────────────────────────┐
│           Incoming Request              │
│    { email?, phoneNumber? }             │
└─────────────────┬───────────────────────┘
                  │
                  ▼
        ┌─────────────────┐
        │  Search DB for  │
        │ matching email  │
        │   OR phone      │
        └────────┬────────┘
                 │
        ┌────────┴────────┐
        │                 │
   No matches        Has matches
        │                 │
        ▼                 ▼
  ┌──────────┐   ┌──────────────────┐
  │  Create  │   │  Find all root   │
  │  new     │   │  primary contacts│
  │ primary  │   │  in matched set  │
  └──────────┘   └────────┬─────────┘
                          │
                 ┌────────┴────────┐
                 │                 │
           1 primary        2+ primaries
                 │                 │
                 │                 ▼
                 │        ┌──────────────┐
                 │        │ MERGE: older │
                 │        │ stays primary│
                 │        │ newer becomes│
                 │        │  secondary   │
                 │        └──────┬───────┘
                 │               │
                 └───────┬───────┘
                         │
                         ▼
               ┌───────────────────┐
               │ Does request have │
               │   NEW info?       │
               └────────┬──────────┘
                        │
               ┌────────┴────────┐
               │                 │
              Yes               No
               │                 │
               ▼                 │
        ┌─────────────┐          │
        │Create new   │          │
        │secondary    │          │
        │contact      │          │
        └──────┬──────┘          │
               │                 │
               └────────┬────────┘
                        │
                        ▼
              ┌──────────────────┐
              │ Build & return   │
              │ consolidated     │
              │ response         │
              └──────────────────┘
```

## 📝 Notes

- The response field is intentionally spelled `primaryContatctId` (with typo) to match the Bitespeed specification exactly
- `phoneNumber` in the request body accepts both `string` and `number` types — it is normalized to a string internally
- The SQLite database is created automatically on first run — no manual setup required
- The database uses **WAL** (Write-Ahead Logging) mode for better concurrent read performance
- Soft-delete is supported via the `deletedAt` column (contacts with a `deletedAt` value are excluded from queries)
